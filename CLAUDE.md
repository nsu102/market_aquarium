다른 문서들은 자동으로 열리지 않는다. **아래 순서를 따라 직접 읽고 사용한다.**

1. **작업을 시작하기 전:** `PLANS.md`를 먼저 읽는다. 없거나 비어 있으면 이번 작업의 계획·리스크를 `PLANS.md`에 먼저 작성한 뒤 사용자 확인을 받고 시작한다. (계획 없이 바로 코딩 금지)
2. **데이터를 만들거나 검증할 때:** `examples/`의 샘플(예: `sample.json`)을 기준 형태로 삼아 그 스키마에 맞춘다.
3. **작업 도중·완료 시:** `CHECKLIST.md`를 갱신한다. 읽은 파일·실행 명령·호출 API를 트레이스로 남긴다.
4. **결과물을 검증·자평할 때:** `EVAL.md`의 하드 게이트(즉시 탈락 조건)를 먼저 확인하고, 통과하면 소프트 페널티/항목별 채점으로 스스로 점검한다.

## 1. 코딩 전에 먼저 생각한다 (Think Before Coding)

- 가정을 말없이 정하지 말 것. **불확실하면 멈추고 질문한다.**
- 해석이 여러 가지로 갈리면 임의로 하나를 고르지 말고 **선택지를 제시**한다. (예: 로그인 → JWT / OAuth / 세션 중 무엇? 먼저 물어본다.)
- 더 단순한 방법이 있으면 말한다. 필요하면 사용자 지시에 반대(push back)한다.

## 2. 단순하게 만든다 (Simplicity First)

- **문제를 푸는 최소한의 코드만** 작성한다.
- 요청하지 않은 기능, 단발성 코드의 추상화, "나중에 쓸지도 모를" 확장성은 넣지 않는다.
- 200줄로 짠 것이 50줄로 가능하면 다시 쓴다.
- 기준: "시니어 개발자가 이걸 보고 과하다고 할까?" → 그렇다면 단순화한다.

## 3. 수술하듯 고친다 (Surgical Changes)

- 고쳐야 할 부분만 건드린다. **옆의 멀쩡한 코드/주석/포맷을 임의로 바꾸지 않는다.**
- 기존 코드 스타일을 따른다(내 취향과 달라도).
- 변경한 모든 줄은 사용자의 요청에 직접 연결되어야 한다.
- 내 변경 때문에 안 쓰이게 된 import·변수만 정리한다. 기존 죽은 코드는 지우지 않고 **언급만** 한다.

## 4. 목표 중심으로 실행한다 (Goal-Driven Execution)

- 모호한 지시를 **검증 가능한 목표**로 바꾼 뒤 실행한다.
  - "버그 고쳐줘" → "버그를 재현하는 테스트 작성 → 실패 확인 → 수정 → 통과 확인"
  - "검증 추가" → "잘못된 입력에 대한 테스트 작성 → 통과시키기"
- 여러 단계 작업은 `PLANS.md`에 짧은 계획을 적고, 각 단계마다 검증 방법을 명시한다.

## 5. 끝났다고 보고하기 전 (정직성 규칙)

- **빌드가 실제로 되는지**, **테스트가 실제로 통과하는지** 직접 실행해 확인한다.
- 실패를 성공으로 보고하지 않는다. 안 되면 안 된다고 적는다.
- 작업 내역(읽은 파일·실행한 명령어·호출한 API)은 `CHECKLIST.md`에 트레이스로 남긴다.
- README와 기본 문서를 작성/갱신한다.

## 6. 권한·환경 제약

- 지정된 작업 폴더 밖의 파일을 수정·삭제하지 않는다.
- 위험한 셸 명령(광범위 삭제, 외부 전송 등)은 실행 전 반드시 확인을 받는다.
- 비밀키·개인정보를 외부에 노출하지 않는다.
